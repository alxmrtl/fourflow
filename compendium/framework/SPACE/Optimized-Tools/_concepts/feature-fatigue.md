# Feature Fatigue

> The documented gap between what people choose before use and what they prefer after use: feature-rich looks appealing in the abstract but creates cognitive overhead in practice that erodes both satisfaction and performance.

## What It Is

Feature fatigue was named and empirically documented by Thompson, Hamilton, and Rust (2005) in a series of studies on consumer technology products. Their finding: before using a product, people reliably prefer the version with more features — more features signal more capability and value. After using a product, however, the preference reverses — the simpler version produces higher satisfaction because the additional features of the complex version created interface complexity, learning costs, and cognitive overhead that outweighed their practical value.

The mechanism they identified is an asymmetry in how people predict and then experience tool use: pre-use evaluation focuses on capability (what can this do?) while in-use experience focuses on effort (how much does it cost me to do what I need?). The value of a rarely-used feature is consistently overestimated before use and the cost of maintaining it in the interface is consistently underestimated.

This finding has been replicated across domain after domain. Dunbar, Moser, and others have extended it to software tools: the features people use most are typically 10-20% of available features; the remaining 80-90% contribute to interface complexity, slower load times, longer learning curves, and the cognitive overhead of maintaining familiarity with a system that is always showing you more than you need.

## Why It Enables Flow

A tool with low feature overhead is one where the interface disappears — where the path from intention to action is direct and requires no navigation, no feature selection, no interface management. This is the precondition for tool embodiment: the tool becomes transparent, an extension of intention rather than an object of attention.

Feature-heavy tools keep themselves visible. They demand interface attention, present choices that must be suppressed, and add a continuous low-level cost to every action. In a flow state — where action and awareness are merging — this surface friction is not just inconvenient; it interrupts the merge. Lean tools protect absorption by demanding nothing from consciousness that isn't the work itself.

## Application

The audit question for any tool in your stack: what percentage of its features do you actually use? If the answer is under 30%, there is likely a leaner alternative that provides the same functional core without the overhead. Preference simplicity for daily-use tools even when the complex version "could do more" — what matters is what you will actually use under load.
